# OpenMC 坐标体系与粒子跟踪全流程深度分析报告

## 上下文

本报告旨在全面解析 OpenMC 粒子跟踪中的坐标机制。目标是建立对 OpenMC 坐标流转的完整认知，为后续新型几何开发奠定基础。报告以通俗语言讲解，辅以具体穿越实例。

## 一、OpenMC 的坐标体系核心概念

### 1.1 什么是"坐标栈"？

想象你站在一栋大楼里。要描述你的精确位置，需要一组嵌套的信息：
- 你在**哪栋楼**的**哪层**的**哪个房间**的**哪个角落**

OpenMC 的坐标栈就是这样的嵌套结构。粒子不是只有一个位置，而是维护一组**从外到内**的坐标层级，每层代表不同参考系下看到的位置。

**最外层（Level 0）** 是全局坐标——就像地球上的经纬度，所有粒子的"绝对位置"都在这里。

**每进入一层嵌套**（一个 Universe 或 Lattice），就增加一层局部坐标——就像从"北京市"进入"海淀区"，坐标系的原点和朝向可能改变。

**最内层** 是粒子实际所在 Material Cell 的坐标——这是物理计算（碰撞、截面查表）真正使用的坐标。

### 1.2 每层坐标里存了什么？

每层（`LocalCoord`）存储以下信息：

| 字段 | 含义 |
|------|------|
| `r` | 该层参考系下的位置向量 (x, y, z) |
| `u` | 该层参考系下的方向向量 (ux, uy, uz) |
| `cell` | 当前所在的 Cell 索引 |
| `universe` | 当前所在的 Universe 索引 |
| `lattice` | 当前所在的 Lattice 索引（如果是在栅格里） |
| `lattice_index` | 栅格内的 (i, j, k) 索引 |
| `rotated` | 该层是否经过旋转变换 |

### 1.3 全局坐标 vs 本地坐标

- **全局坐标** `r()` = `coord[0].r` —— Level 0 的位置，永远在 Root Universe 的参考系中
- **本地坐标** `r_local()` = `coord[n-1].r` —— 最内层的位置，在粒子当前 Material Cell 所属 Universe 的参考系中

它们描述的是**同一个物理点**，只是"站在不同的观察者角度"看。

### 1.4 嵌套层级示例

```
Root Universe (全局坐标系)
  └─ Cell "反应堆容器" (填充 = Universe "堆芯", 有平移和旋转)
       └─ Universe "堆芯" (局部坐标系，经过平移+旋转)
            └─ Cell "燃料组件区域" (填充 = Lattice "组件栅格")
                 └─ Lattice "组件栅格" (栅格局部坐标系)
                      └─ Cell "燃料棒" (填充 = Material UO2) ← 物理计算在这里
```

这个例子有 3 层坐标：全局 → 堆芯局部 → 栅格局部。坐标栈的大小 `n_coord = 3`。

## 二、坐标变换的原理

### 2.1 进入 Universe 填充时的变换

当一个 Cell 的填充物是另一个 Universe 时，需要把坐标从外层参考系变换到内层参考系。变换分两步：

**第一步：平移** —— 减去 Cell 定义的平移向量 `translation`

> 直觉理解：就像把地图的原点从城市中心移到了某个小区的中心。你相对于小区的位置 = 你相对于市中心的位置 − 小区相对于市中心的位置。

**第二步：旋转** —— 用 Cell 定义的旋转矩阵 R 变换位置和方向

> 直觉理解：就像小区的朝向和城市的主轴不一致（小区是斜着建的）。你需要把你的位置和行进方向都转到小区的参考系里。

**关键点**：旋转必须同时作用于位置和方向。因为如果你只旋转了位置但方向没变，下一步移动时方向就不对了。

### 2.2 进入 Lattice 填充时的变换

Lattice 比 Universe 多了一步——需要先确定粒子在栅格的哪个"格子"里：

1. **先做和 Universe 相同的变换**（平移 + 旋转）
2. **计算栅格索引** `get_indices(r, u)` —— 根据变换后的位置，确定在哪个 (i, j, k) 格子
3. **转换到格子内的局部坐标** `get_local_position(r, i_xyz)` —— 把坐标原点移到格子中心

> 直觉理解：Lattice 就像一张棋盘。先确定棋子在哪一格（索引），再描述棋子在格内的偏移（局部坐标）。格子中心就是局部坐标的原点。

### 2.3 矩形栅格（RectLattice）的坐标计算

矩形栅格是最简单的情况：

- **栅格索引**：`(位置 − 左下角) / 格距`，然后取整
- **局部坐标**：`位置 − (左下角 + (索引 + 0.5) × 格距)`
- 局部坐标原点在格子**中心**，所以有 `+0.5`

如果粒子恰好在格子的边界上，就用运动方向来决定属于哪一侧——粒子往哪边走，就归哪边。

### 2.4 六角形栅格（HexLattice）的坐标计算

六角形更复杂，因为格元不是矩形排列，而是像蜂巢一样交错：

- 先把位置转换到一个"斜坐标系"（涉及 √3 因子）
- 计算出粗略的索引
- 然后用"距离最小化"在 4 个候选格元中选最近的一个
- 如果距离相同（恰好在边界），用运动方向来判定

## 三、粒子跟踪的完整生命周期

### 3.1 全流程概览

一个粒子从产生到消亡，经历以下循环：

```
  ┌──────────────────────────────────────────────────────┐
  │                                                      │
  │  1. 采样产生粒子（位置、方向、能量）                   │
  │         │                                            │
  │         ▼                                            │
  │  2. 定位：粒子在哪个 Cell？（构建坐标栈）              │
  │         │                                            │
  │         ▼                                            │
  │  3. 计算到最近边界的距离                              │
  │         │                                            │
  │         ▼                                            │
  │  4. 采样到下次碰撞的距离                              │
  │         │                                            │
  │         ▼                                            │
  │  5. 比较并取较小值：移动到碰撞点或边界                 │
  │         │                                            │
  │     ┌───┴───────────────┐                            │
  │     │                   │                            │
  │  碰撞发生          到达边界                           │
  │     │                   │                            │
  │     ▼                   ▼                            │
  │  6a. 物理碰撞处理   6b. 边界穿越处理                  │
  │  （改变方向/能量    （更新坐标栈，                     │
  │   可能产生新粒子     找到新 Cell）                    │
  │     │                   │                            │
  │     └───────┬───────────┘                            │
  │             │                                        │
  │             ▼                                        │
  │  7. 粒子还活着吗？                                   │
  │     │           │                                    │
  │    是          否                                    │
  │     │           │                                    │
  │  回到第3步    结束（被吸收/逃逸/俄罗斯轮盘杀掉）        │
  │                                                      │
  └──────────────────────────────────────────────────────┘
```

### 3.2 第一步：粒子定位（找 Cell）

**这是坐标栈的构建过程。**

当粒子刚产生或从边界穿越回来，OpenMC 需要知道粒子在哪。调用 `exhaustive_find_cell()`：

1. 从 Root Universe 开始，用全局坐标逐个检查每个 Cell 的区域条件（是否在某组表面的内侧/外侧）
2. 找到包含该点的 Cell 后，看它的填充物是什么：
   - **Material** → 找到了！设置材料属性，结束
   - **Universe** → 做坐标变换（平移+旋转），在新 Universe 中继续搜索
   - **Lattice** → 做坐标变换 + 确定栅格索引 + 转换格内局部坐标，在对应格子的 Universe 中继续搜索
3. 重复直到找到 Material Cell

每深入一层，`n_coord` 加一，坐标栈就多一层。

### 3.3 第二步：计算到边界的距离

坐标栈的每一层都可能有边界：

- **Cell 表面** —— 由平面、圆柱、球面等曲面定义。每层 Cell 都有自己的表面距离
- **Lattice 边界** —— 栅格格子的边界。只有在 lattice 层级才有

OpenMC 逐层扫描，计算每一层到最近边界的距离，取**所有层级中的最小值**。同时记录这个最小距离出现在哪一层、是什么类型的边界。

### 3.4 第三步：移动粒子

移动时，**坐标栈的所有层级同步更新**：

每一层都执行 `r += 距离 × u`（用各自层级自己的方向向量）。

> 为什么每一层都要更新？因为每层的方向向量可能不同（经过旋转），所以每层需要用自己的 u 来计算移动后的位置。这保证了所有层级始终指向同一个物理点。

### 3.5 第四步：碰撞处理（物理事件）

如果碰撞距离更短，粒子在碰撞点发生物理交互（散射、吸收、裂变等）。碰撞可能改变粒子的方向和能量。方向改变后，坐标栈中所有层级的方向都要**同步更新**。

### 3.6 第五步：边界穿越处理

**这是坐标栈维护最复杂的部分。**

到达边界时，需要：
1. **保存旧位置信息**（记录刚才在哪个 Cell）
2. **截断坐标栈**到穿越发生的层级（更内层的坐标不再有效）
3. **根据边界类型分两种处理**：

#### 情况 A：普通表面穿越（Cell 表面）

粒子从一个 Cell 穿到相邻 Cell。由于只改变了最内层或某一层，可以在附近搜索新的 Cell（邻居搜索），通常不需要从头重建整个坐标栈。

#### 情况 B：Lattice 边界穿越

粒子从一个栅格格子穿到相邻格子。此时需要：
1. 更新栅格索引（比如从 (3,2) 变成 (4,2)）
2. 从**上层坐标**重新计算新格子内的局部坐标（因为每个格子的原点不同）
3. 在新格子的 Universe 中搜索 Cell

如果新索引超出栅格范围（粒子走出了 Lattice），就需要把坐标栈**完全重置**到 Level 0，从全局坐标从头搜索。

还有一种特殊情况叫**角落穿越（corner crossing）**：粒子从 Lattice 的一个角穿出去，在新格子中找不到 Cell。此时也要重置坐标栈从头搜索。

## 四、具体穿越实例

### 4.1 场景设置

假设有一个简化的燃料组件模型：

```
全局坐标空间 (Root Universe)
└── Cell "水池" (填充 = Universe "组件容器", 平移 = (10, 5, 0))
    └── Universe "组件容器" (局部坐标，原点在全局 (10, 5, 0))
        └── Cell "栅格区域" (填充 = RectLattice, 无旋转)
            └── RectLattice (pitch = 2×2, lower_left = (−4, −4, 0))
                ├── 格子 (0,0): Universe "燃料" → Material UO2
                ├── 格子 (1,0): Universe "燃料" → Material UO2
                ├── 格子 (0,1): Universe "燃料" → Material UO2
                └── 格子 (1,1): Universe "水"   → Material H2O
```

### 4.2 粒子产生

粒子在全局坐标 (11.0, 5.5, 0.0) 产生，方向 (1, 0, 0)（沿 +X 飞行）。

### 4.3 定位过程（坐标栈构建）

**Level 0**：在 Root Universe 中搜索
- 检查 Cell "水池"的区域 → 包含点 (11.0, 5.5, 0.0) ✓
- 这个 Cell 填充的是 Universe "组件容器"，有平移 (10, 5, 0)
- **坐标变换**：新位置 = (11.0, 5.5, 0.0) − (10, 5, 0) = (1.0, 0.5, 0.0)
- `n_coord = 2`

**Level 1**：在 Universe "组件容器" 中搜索
- 检查 Cell "栅格区域" → 包含点 (1.0, 0.5, 0.0) ✓
- 这个 Cell 填充的是 RectLattice
- **计算栅格索引**：lower_left = (−4, −4, 0), pitch = (2, 2)
  - i = floor((1.0 − (−4)) / 2) = floor(2.5) = 2... 不在 0..1 范围
  - 实际上这个位置在栅格外... 让我们换个更合理的位置

让我用一组自洽的数值重新演示：

### 4.4 修正后的实例

```
全局坐标空间 (Root Universe)
└── Cell "反应堆" (填充 = Universe "堆芯", 平移 = (10, 10, 0), 无旋转)
    └── Universe "堆芯"
        └── Cell "栅格区" (填充 = RectLattice)
            └── RectLattice (pitch = 2×2, lower_left = (0, 0, 0), 共 3×3 格子)
                ├── 格子 (0,0) ~ (2,2): 各自填充不同 Universe
                └── 例如格子 (1,1) 的中心在全局 (10+1*2+1, 10+1*2+1, 0) = (13, 13, 0)
```

**粒子产生位置**：全局 (12.5, 10.5, 0.0)，方向 (1, 0, 0)

#### 定位过程

**Level 0**：Root Universe，全局坐标 (12.5, 10.5, 0.0)
- 找到 Cell "反应堆"，填充 = Universe "堆芯"，平移 = (10, 10, 0)
- 变换：r = (12.5, 10.5, 0.0) − (10, 10, 0) = **(2.5, 0.5, 0.0)**
- 方向不变：(1, 0, 0)
- `n_coord = 2`

**Level 1**：Universe "堆芯"，局部坐标 (2.5, 0.5, 0.0)
- 找到 Cell "栅格区"，填充 = RectLattice (lower_left = (0,0,0), pitch = (2,2))
- 先做 Cell 自身变换（此处无平移旋转，保持不变）
- **计算栅格索引**：
  - i = floor((2.5 − 0) / 2) = floor(1.25) = **1**
  - j = floor((0.5 − 0) / 2) = floor(0.25) = **0**
- **计算格内局部坐标**：
  - r_local.x = 2.5 − (0 + (1 + 0.5) × 2) = 2.5 − 3.0 = **−0.5**
  - r_local.y = 0.5 − (0 + (0 + 0.5) × 2) = 0.5 − 1.0 = **−0.5**
- 粒子在格子 (1, 0) 内的局部坐标 (−0.5, −0.5, 0.0)
- `n_coord = 3`

**Level 2**：格子 (1, 0) 对应的 Universe 中搜索
- 假设这个 Universe 里有一个 Cell "燃料棒"，包含点 (−0.5, −0.5, 0.0) ✓
- 填充 = Material "UO2"
- **找到 Material！定位完成。**

此时坐标栈状态：

| 层级 | 位置 r | 方向 u | Cell | Universe | 说明 |
|------|--------|--------|------|----------|------|
| Level 0 | (12.5, 10.5, 0.0) | (1, 0, 0) | "反应堆" | Root | 全局坐标 |
| Level 1 | (2.5, 0.5, 0.0) | (1, 0, 0) | "栅格区" | "堆芯" | 堆芯局部坐标 |
| Level 2 | (−0.5, −0.5, 0.0) | (1, 0, 0) | "燃料棒" | 格子(1,0)的Univ | 格内局部坐标 |

#### 移动和碰撞

假设采样到下次碰撞距离为 0.8 cm。所有层级同步移动：

| 层级 | 移动前 | 移动后 (r += 0.8 × u) |
|------|--------|----------------------|
| Level 0 | (12.5, 10.5, 0) | (13.3, 10.5, 0) |
| Level 1 | (2.5, 0.5, 0) | (3.3, 0.5, 0) |
| Level 2 | (−0.5, −0.5, 0) | (0.3, −0.5, 0) |

粒子在 Level 2 的位置从 −0.5 变到 0.3，但格子 (1,0) 的范围是 [−1, 1]（中心在原点，宽度 2），所以还在格子内。碰撞在格内发生。

#### Lattice 边界穿越

假设碰撞后粒子方向变为 (1, 0, 0)，新碰撞距离是 1.5 cm。

但首先计算到边界的距离：在 Level 2，粒子在格内 x = 0.3，格子右边界在 x = 1.0（pitch/2 = 1.0），方向 +x，所以到 lattice 边界的距离 = (1.0 − 0.3) / 1.0 = **0.7 cm**。

0.7 < 1.5，所以粒子先到达 lattice 边界！

**移动 0.7 cm**（所有层级同步）：

| 层级 | 移动前 | 移动后 |
|------|--------|--------|
| Level 0 | (13.3, 10.5, 0) | (14.0, 10.5, 0) |
| Level 1 | (3.3, 0.5, 0) | (4.0, 0.5, 0) |
| Level 2 | (0.3, −0.5, 0) | (1.0, −0.5, 0) |

**边界穿越处理**：

1. 这是 Lattice 边界，lattice_translation = (1, 0, 0)
2. 更新栅格索引：(1, 0) → **(2, 0)**
3. 从 Level 1 重新计算格内局部坐标：
   - Level 1 位置 = (4.0, 0.5, 0)
   - 格子 (2, 0) 的中心在 Level 1 坐标系中 = (0 + (2+0.5)×2, 0 + (0+0.5)×2, 0) = (5.0, 1.0, 0)
   - r_local = (4.0 − 5.0, 0.5 − 1.0, 0) = **(−1.0, −0.5, 0)**
4. 检查索引 (2, 0) 是否有效：3×3 栅格，0 ≤ 2 < 3 ✓
5. 在格子 (2, 0) 的 Universe 中搜索 Cell → 找到 Material → 继续

注意：粒子到达边界时 Level 2 的 x 恰好等于 1.0（格子右边界），穿过去之后在新格子里 x = −1.0（格子左边界）。这是连续的——边界对两侧是同一个面。

#### 如果粒子走出栅格

假设粒子继续沿 +x 飞行，从格子 (2, 0) 穿到索引 (3, 0)——但栅格只有 3 列（索引 0, 1, 2），所以 3 是无效索引。

此时 OpenMC 会：
1. 将 `n_coord` 重置为 1（只保留 Level 0 全局坐标）
2. 从全局坐标 (14.0+移动距离, 10.5, 0) 重新做 exhaustive_find_cell
3. 在 Root Universe 中搜索包含该点的新 Cell

这就是"坐标栈重置"——在 OpenMC 中是最后的兜底手段，代价较高但保证正确性。

## 五、坐标变化的完整触发条件总结

### 5.1 坐标栈构建（增加层级）

| 事件 | 坐标变化 |
|------|----------|
| 进入填充 Universe 的 Cell | 新增一层：平移 → 旋转 |
| 进入填充 Lattice 的 Cell | 新增一层：平移 → 旋转 → 计算索引 → 格内局部坐标 |
| 找到 Material Cell | 不新增层级，定位完成 |

### 5.2 坐标栈维护（移动和穿越）

| 事件 | 坐标变化 |
|------|----------|
| 粒子移动（碰撞或向边界飞行） | 所有层级的位置同步更新：`r += d × u` |
| 碰撞改变方向 | 最内层方向改变，后续由坐标变换保证一致性 |
| Cell 表面穿越 | 截断到穿越层级，局部搜索新 Cell |
| Lattice 边界穿越（索引有效） | 更新索引，从上层重新计算格内坐标，局部搜索 |
| Lattice 边界穿越（索引无效） | 重置到 Level 0，从头搜索 |
| 角落穿越（Corner crossing） | 重置到 Level 0，从头搜索 |

### 5.3 坐标栈重置的代价

从头搜索（`exhaustive_find_cell`）需要遍历 Root Universe 的所有 Cell 并重建整个坐标栈，代价较高。OpenMC 尽量避免这种情况，只在必要时使用：

- 普通表面穿越 → 用邻居搜索（快速）
- Lattice 内部穿越 → 用索引增量 + 局部搜索（快速）
- **只有出了 Lattice 或遇到角落才需要从头搜索**（兜底）

---

## 六、对新型几何开发的启示

### 6.1 如果你要开发一种新型填充类型

需要在以下位置加入新逻辑：

1. **`find_cell_inner()`** —— 定义新型填充时的坐标变换规则
2. **`distance_to_boundary()`** —— 定义到新型边界的距离计算
3. **边界穿越处理** —— 定义穿越新型边界后如何更新坐标栈

### 6.2 坐标一致性的"契约"

无论你开发什么新型几何，必须遵守以下约定：

1. **所有层级同步移动** —— `move_distance` 遍历所有层，每层用各自的 (r, u)
2. **旋转同时作用于位置和方向** —— 不能只转一个
3. **最内层局部坐标必须使 Cell 的包含判断正确** —— `Cell::contains(r_local, u_local)` 必须返回 true
4. **穿越后坐标栈必须能重建** —— 截断后重新搜索必须成功

### 6.3 可直接复用的基础设施

| 组件 | 所在文件 | 用途 |
|------|----------|------|
| `LocalCoord` | particle_data.h | 坐标层级数据结构，直接复用 |
| `Position::rotate()` | position.h | 旋转变换 |
| `Position::inverse_rotate()` | position.h | 逆向旋转（正交矩阵性质） |
| `BoundaryInfo` | geometry.h | 边界穿越信息载体 |
| `exhaustive_find_cell()` | geometry.cpp | 兜底的完整坐标栈重建 |
| `model::n_coord_levels` | model.h | 系统最大坐标深度 |
